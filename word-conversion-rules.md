# Word 转换规则

## Rule: 避免使用加粗格式和特殊符号

面向 Word 交付的中文方案文档，正文中不应使用 Markdown 加粗语法（双星号包裹）来强调内容。加粗格式在复制到 Word 后会导致排版不统一，且不便于后续样式调整。如需强调某个术语或关键概念，应通过上下文语境或完整的句子表达来体现，而不是依赖加粗格式。

同时，正文中不应使用表情符号（如 ✅ ❌ ⚠️ ✓ ✗）或其他特殊 Unicode 符号。这些符号在 Word 中可能无法正常显示，或与正文字体风格不一致。应使用文字描述替代，例如将"✅ 可用"改写为"可用"，将"❌ 不可用"改写为"不可用"。

## Rule: 清理正文中的 Markdown 内联代码

面向 Word 交付的中文方案文档，应避免在最终正文中保留 Markdown 内联代码格式。对于寄存器名、接口名、字段名、章节引用、配置项、路径和命令等技术标识，在编写中的 Markdown 草稿里可以阶段性保留反引号，以帮助作者区分技术对象；但在进入 Word 交付整理阶段时，正文中这些内容应统一改写为普通文本。

被反引号包裹的章节号或章节引用，应按中文技术文档习惯补足语义，改写为完整的章节引用表达。参与 Word 正文转换的段落、表格标题、图题、图注、章节标题和结论段中，不应残留单反引号内联代码。

## Rule: 只有 TODO 清单允许保留 Markdown 辅助写法

面向后续 Word 交付的方案文档中，只有 TODO 清单可以明确保留 Markdown 辅助写法，例如反引号、待办式占位、临时强调、路径提示或其他便于继续编写和跟踪的问题记录。这样做的原因是 TODO 清单本身属于编写管理区域，而不是最终交付正文。

除 TODO 清单外，正文其他区域默认不允许继续依赖 Markdown 辅助写法来表达正式内容。变更记录、源码片段或纯技术附录若确有保留 Markdown 原貌的必要，应在文档中明确标识其不属于 Word 正文交付内容；否则均按最终正文标准处理。

## Rule: 正文优先采用整段文字表达

面向 Word 交付的正文内容，应优先采用一段一段的连续文字展开说明，而不是频繁使用无序列表、碎片化短句或过多层级的小点。每一段应围绕一个明确主题展开，例如设计目标、接口职责、状态变化条件、寄存器作用范围或模块边界，使内容在复制到 Word 后仍能保持自然的段落结构，减少人工重新排版的工作量。

当某部分内容既可以写成列表，也可以写成顺畅段落时，默认优先选择段落表达。只有在枚举项本身具有明显并列关系，且不用列表会降低可读性时，才使用列表形式。

## Rule: 谨慎使用无序列表和多级列表

无序列表在 Markdown 中书写方便，但转换到 Word 后往往容易出现缩进、间距和编号层级不统一的问题。因此，正文中应尽量少用无序列表，尤其要避免连续出现多层嵌套列表。对于可以通过“首先、其次、最后”“包括”“分别为”等自然语言连接起来的内容，应直接写成完整句子或连续段落。

如果确实需要列出若干项目，优先使用短而平的单层结构，并控制条目数量，避免在一个小节内堆叠多组列表。章节主体不应依赖列表来支撑逻辑，而应以段落叙述为主。对于具有明显先后顺序的操作步骤、处理流程或验证流程，优先使用有序编号或连续段落，而不是无序列表。

## Rule: 表格内容保持简洁，便于后续 Word 排版

需要保留表格时，应尽量让表格承担“索引、对照、归类、摘要”的作用，而不要把大段解释文字塞进单元格。表头名称应直接、稳定，单元格内容应简短清楚，适合作为后续 Word 中的规范表格内容。对于需要较长解释的事项，应先在表格外用段落说明，再在表格中保留简要结论。

寄存器表、接口表、模块职责表、术语对照表等适合保留表格形式，但每个单元格都应避免过长句子、过多括号说明和多层并列描述。若某项说明超出一两句话，通常更适合改回表格前后的正文段落。

当表格本身承载的是字段定义、位定义或消息记录格式时，允许保留必要的技术密度，但仍应遵循“字段放表内，解释放表外”的原则。对于列数过多、单元格说明过长或连续跨越多个页面的表格，应优先拆分为“引导段落 + 多个小表 + 补充说明”的结构，而不是继续扩张单一大表。

## Rule: 章节内容按 Word 正文阅读习惯组织

文档在编写时应默认面向“顺序阅读和人工排版”场景，而不是只面向 Markdown 预览。章节内部宜采用“先总述、后展开”的写法，先用一段话说明本节目的、适用范围和边界，再进入必要的表格或局部分项说明。这样转换到 Word 后，即使样式尚未完全整理，读者也能先通过正文段落理解上下文。

对于接口定义、寄存器定义、模块设计这类容易写成“条目堆叠”的内容，也应尽量补足承上启下的自然语言说明，避免整个章节只剩标题、表格和零散短句。

对于位定义、接口定义和消息格式定义等必须大量使用表格的章节，表格前应至少有一段交代本节职责边界、适用范围或组织方式的引导文字；表格后如存在特殊约束、例外情况或读取规则，也应补充一段总结文字，避免形成只有表格没有上下文的页面。

## Rule: 图示与 Mermaid 只作为草稿占位，交付前应替换为正式插图

Markdown 中的 Mermaid、ASCII 流程图或简化结构图，可以作为方案编写阶段的占位表达，用于帮助确认系统分层、模块关系和流程方向。但面向 Word 交付时，不应把 Mermaid 源码直接作为正文内容保留，而应替换为正式整理后的图片、Visio 图或统一风格的结构图。

在使用占位图示时，图前应有引导说明，图后应有一段对关键关系的解释，避免读者只能依赖图本身理解设计。图题、图号和图内命名应在转 Word 阶段统一整理。

## Rule: 控制标题层级和单节碎片化程度

Markdown 文档虽然可以轻易增加很多层标题，但 Word 正文阅读和排版通常不适合过深层级。方案类文档应尽量控制标题层级，避免为了容纳零散信息不断向下细分到过细的小节。若某小节只有一两句话或只有一个小表，更适合并回上一级小节，通过段落或子表表达，而不是单独增加一层标题。

在寄存器、接口和模块说明章节中，应优先通过”总述段落 + 若干顺序小节 + 表格”的方式组织，而不是持续堆叠过深的小标题树。

标题层级限制在三级以内（对应 Markdown 的 `#` `##` `###`），不使用四级及以下标题。若某小节内容过少，应并回上级小节，通过段落或子表表达。

## Rule: Markdown 结构服务于后续 Word，而不是反过来

编写 Markdown 时，应有意识地控制那些会给 Word 转换带来额外整理成本的格式，例如层级过深的标题、过密的列表、过长的表格、依赖 Markdown 特性的强调写法等。对于面向正式交付的方案类文档，应把“复制到 Word 后容易继续整理和套模板”作为优先目标，而不是追求 Markdown 本身的花哨结构。

如果某种 Markdown 写法在预览里看起来整齐，但复制到 Word 后通常需要大量手工修正，则应优先改写为更朴素、更接近普通中文技术文档的表达方式。
