# 中文技术文档用词规范

本文件记录项目文档中应避免的非标准用词，以及对应的正确写法。适用于所有 `.md` 文档、注释和提交说明。

## 禁用词替换表

| 禁用 | 正确替代 | 说明 |
|------|----------|------|
| 收敛 | 确定、稳定、完善 | "收敛"是数学/优化领域术语，普通技术文档用"确定"即可 |
| 固化 | 定义、确定、规定、写入 | "固化"过于抽象，直接说"定义了什么"更清楚 |
| 主口径 | 权威依据、主要参考文档 | "口径"是宣传用语，技术文档用"依据"或"参考" |
| 叙事 | 描述、内容、内容线 | "叙事"是文学用语，技术文档不需要 |
| 落 RTL / 落地 | 编写 RTL、实现 | "落"作动词是口语省略，正式文档应写完整 |
| 可读化 | 添加注释、提高可读性 | "可读化"不是标准说法，应说明具体做了什么 |
| 收口 | 收尾、确认、关闭 | "收口"是管理口语，技术文档用"收尾" |
| 流水账 | 冗余过程记录 | 口语化，正式文档应避免 |
| 堆（抽象/职责） | 叠加、增加、堆积 | "堆"作动词过于口语 |
| 脏（工作区） | 不够干净、存在未整理内容 | "脏"是程序员俚语 |

## 原则

1. **说人话**：技术文档的目的是让人看懂。用最直接的词把事情说清楚，避免展示文采式表达。
2. **不用比喻代替术语**：如果有一个准确的技术词，就用它；如果没有，用白话解释，不要用另一个领域的术语来借喻。
3. **动词要完整**："确定""定义""编写""实现"都是完整动词，不要省略成"落""收"。
4. **名词要具体**：说"主要参考文档"比说"主口径"清楚十倍。
5. **优先使用正向表达**：避免使用"不是……而是……""不在……而在……""应……不应……""只……而不……"这类否定转折句式。直接说明对象的职责、重点或行为，再补充边界限制。
6. **唯一例外**：已成为 FPGA 领域标准术语的用法可以保留，例如"时序收敛"（timing closure）。

## 设计文档语气规范

适用于设计性文档（设计方案、概要设计、详细设计等）。设计文档描述"是什么"和"怎么做"，不描述"应该怎么做"。

### 避免规范性"应"语气

设计文档的正文不使用"应""不应""应当""须"等规范性动词。这些词属于需求规格或规范文档的语气，设计文档直接陈述设计事实。

| 写法 | 问题 | 改法 |
|------|------|------|
| "单路 IP 核应支持 BC、RT、BM 三种角色" | 规范性语气 | "单路 IP 核支持 BC、RT、BM 三种角色" |
| "模块划分应清晰" | 规范性语气 | "模块划分清晰" |
| "硬件应在空闲后生效" | 规范性语气 | "硬件在空闲后生效" |
| "由软件侧完成换算" | 无问题 | 保持 |

唯一例外：TODO 清单允许保留规范性语气，因为 TODO 属于编写管理区域，不是交付正文。

### 避免否定转折句式

不用"是……而不是……""只承担……而不混入……""不应仅停留在……而应在……""只负责……而不承担……"等写法。直接说明设计做了什么，需要限定边界时另起一句或用"仅""只"等词正面限定。

| 写法 | 问题 | 改法 |
|------|------|------|
| "采用 A 方式，而不是依赖 B" | 否定转折 | "采用 A 方式，共享同一物理基础" |
| "只负责 X，而不承担 Y" | 否定转折 | "负责 X，Y 由其他模块承担" |
| "不应仅停留在汇总层，而应在模块中定位" | 否定转折 | "在错误产生的对应模块中完成定位" |
| "记录流仍应保持……原则，而不应因为……" | 否定转折 | "记录流保持……原则，消息不完整时仍然写入" |
| "面向的是 X 而不是 Y" | 否定转折 | "面向 X，与 Y 相区分" |
| "而不是把端口命名完全确定" | 否定转折 | "端口命名在详细设计阶段再进一步确定" |
