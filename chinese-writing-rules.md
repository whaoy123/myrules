# 中文技术文档用词规范

本文件记录项目文档中应避免的非标准用词、表达习惯和行文规则。适用于所有 `.md` 文档、注释和提交说明。

## 禁用词替换表

| 禁用 | 正确替代 | 说明 |
|------|----------|------|
| 收敛 | 确定、稳定、完善 | "收敛"是数学/优化领域术语，普通技术文档用"确定"即可 |
| 固化 | 定义、确定、规定、写入 | "固化"过于抽象，直接说"定义了什么"更清楚 |
| 主口径 | 权威依据、主要参考文档 | "口径"是宣传用语，技术文档用"依据"或"参考" |
| 叙事 | 描述、内容、内容线 | "叙事"是文学用语，技术文档不需要 |
| 落 RTL / 落地 | 编写 RTL、实现 | "落"作动词是口语省略，正式文档应写完整 |
| 可读化 | 添加注释、提高可读性 | "可读化"不是标准说法，应说明具体做了什么 |
| 收口 | 收尾、确认、关闭 | "收口"是管理口语，技术文档用"收尾" |
| 流水账 | 冗余过程记录 | 口语化，正式文档应避免 |
| 堆（抽象/职责） | 叠加、增加、堆积 | "堆"作动词过于口语 |
| 脏（工作区） | 不够干净、存在未整理内容 | "脏"是程序员俚语 |

## 原则

1. **说人话**：技术文档的目的是让人看懂。用最直接的词把事情说清楚，避免展示文采式表达。
2. **不用比喻代替术语**：如果有一个准确的技术词，就用它；如果没有，用白话解释，不要用另一个领域的术语来借喻。
3. **动词要完整**："确定""定义""编写""实现"都是完整动词，不要省略成""落""收"。
4. **名词要具体**：说"主要参考文档"比说"主口径"清楚十倍。
5. **直接推进论点**：每一句都应推进论点或提供必要信息，不绕弯、不防御、不为规避禁用句式而加入无必要的解释。不能给第一次阅读的读者增加理解负担。
6. **唯一例外**：已成为 FPGA 领域标准术语的用法可以保留，例如"时序收敛"（timing closure）。

## 直述推进与自然衔接

本节约束行文的推进方式和句间衔接。核心要求：每一段有明确任务，直接展开，不迂回、不防御、不重复。

### 禁用迂回立论

正文不得使用"与其说 A 不如说 B""某种意义上""从某种角度来看""换个角度看"等迂回开头。论点直接陈述，论据紧跟其后。

| 写法 | 问题 | 改法 |
|------|------|------|
| "与其说这是模块划分，不如说这是一种职责分配" | 迂回立论 | "这是按职责进行的模块划分" |
| "从某种角度来看，这种设计兼顾了性能和面积" | 迂回立论 | "这种设计兼顾了性能和面积" |
| "某种程度上，该方案满足了时序要求" | 防御式表达 | "该方案满足了时序要求" |

### 禁用防御式表达

不在正文使用"不得不承认""客观地说""坦率地说""实事求是地讲""不可否认""毋庸置疑"等防御式插入语。这些表达暗示作者心虚或预设读者反对，技术文档不需要。

### 禁用滥用的英文衔接句式

中文技术文档中不得机械套用以下英文行文模式。这些模式在英文中常见但翻译成中文后往往冗余、迂回或打断节奏。

| 滥用句式 | 问题 | 改法 |
|----------|------|------|
| "not X but Y" / "不是 X 而是 Y" | 否定转折 | 直接说 Y，需要区分时另起一句 |
| "rather than X, Y" / "与其 X，不如 Y" | 迂回 | 直接说 Y |
| "since X, Y" / "由于 X，因此 Y" | 双重因果 | 保留一个因果连接词即可 |
| "however" / "然而，但是" | 滥用转折 | 大多数情况直接接下一句，无需转折词 |
| "therefore" / "因此，所以" | 滥用因果 | 上下文已经明确因果关系时省略 |
| "not only X but also Y" / "不仅 X 而且 Y" | 否定叠加 | 直接并列："X 和 Y""X，同时 Y" |
| "it is worth noting that" / "值得注意的是" | 冗余引导 | 直接说值得说的内容 |
| "in fact" / "事实上" | 虚假转折 | 删除，直接陈述事实 |

### 方法选择要自然衔接

当正文讨论一种设计选择或实现方式时，需要从背景自然过渡到结论，不能前一句还在讲多种可能性，下一句突然跳到当前设定而不交代选择理由。

| 写法 | 问题 | 改法 |
|------|------|------|
| "实现该功能有三种方式。本设计采用方式 A。" | 突然跳转 | "实现该功能有三种方式。方式 A 的优势是……，因此本设计采用方式 A。" |
| "寄存器可以按字或按位访问。寄存器定义如下表。" | 缺衔接 | "寄存器可以按字或按位访问。考虑到总线宽度统一，本设计按字访问。寄存器定义如下表。" |

衔接不需要长篇大论，一两句话交代选择依据即可。目的是让读者理解"为什么是这个"，而不是只看到"是这个"。

### 技术细节不堆砌

正文开头先说明对象是什么、做什么用，再展开内部实现细节。不要在第一段就抛出寄存器地址、位域宽度、时序参数等底层信息。

| 写法 | 问题 | 改法 |
|------|------|------|
| "该模块基地址 0x4000，包含 12 个 32 位寄存器，支持 BC/RT/BM 三种模式……" | 细节堆砌 | "该模块实现 1553B 总线协议处理，支持 BC、RT、BM 三种模式。模块基地址 0x4000，内部包含 12 个 32 位寄存器。" |

### 每段有明确任务，不重复

一个段落只解决一个问题。段落之间不重复同一信息。如果前一段已经说明了某个结论，后续段落直接引用或推进，不再重新解释一遍。

## 设计文档语气规范

适用于设计性文档（设计方案、概要设计、详细设计等）。设计文档描述"是什么"和"怎么做"，不描述"应该怎么做"。

### 避免规范性"应"语气

设计文档的正文不使用"应""不应""应当""须"等规范性动词。这些词属于需求规格或规范文档的语气，设计文档直接陈述设计事实。

| 写法 | 问题 | 改法 |
|------|------|------|
| "单路 IP 核应支持 BC、RT、BM 三种角色" | 规范性语气 | "单路 IP 核支持 BC、RT、BM 三种角色" |
| "模块划分应清晰" | 规范性语气 | "模块划分清晰" |
| "硬件应在空闲后生效" | 规范性语气 | "硬件在空闲后生效" |
| "由软件侧完成换算" | 无问题 | 保持 |

唯一例外：TODO 清单允许保留规范性语气，因为 TODO 属于编写管理区域，不是交付正文。

### 避免否定转折句式

不用"是……而不是……""只承担……而不混入……""不应仅停留在……而应在……""只负责……而不承担……"等写法。直接说明设计做了什么，需要限定边界时另起一句或用"只""仅"等词正面限定。

| 写法 | 问题 | 改法 |
|------|------|------|
| "采用 A 方式，而不是依赖 B" | 否定转折 | "采用 A 方式，共享同一物理基础" |
| "只负责 X，而不承担 Y" | 否定转折 | "负责 X，Y 由其他模块承担" |
| "不应仅停留在汇总层，而应在模块中定位" | 否定转折 | "在错误产生的对应模块中完成定位" |
| "记录流仍应保持……原则，而不应因为……" | 否定转折 | "记录流保持……原则，消息不完整时仍然写入" |
| "面向的是 X 而不是 Y" | 否定转折 | "面向 X，与 Y 相区分" |
| "而不是把端口命名完全确定" | 否定转折 | "端口命名在详细设计阶段再进一步确定" |
